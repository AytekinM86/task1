1. What this app does

Imagine you just built an empty hospital database with tables for patients, doctors, appointments, and prescriptions. To test it, demo it, or build reports on top of it, you need data — but typing thousands of rows by hand is impossible, and the rows have to make sense together (an appointment must point to a real patient and a real doctor; a prescription only exists for an appointment that actually happened).

This app generates all of that automatically. In one command it:

  • 🎲 Invents believable people, doctors, departments, medications, visits, and prescriptions using the Faker library.
  • 🔗 Links them correctly so every foreign key points at something real.
  • 🕒 Keeps time consistent — past visits are "completed/cancelled", future visits are "scheduled", and a prescription is always issued after its appointment.
  • 💾 Inserts everything in one transaction — it all lands, or nothing does (no half-filled database).
  • ✅ Respects every database rule (text length limits, allowed values, unique emails, etc.).
Mental model: it's a robot data-entry clerk that already knows the rules of the hospital and never makes a typo or contradicts itself.

2. The big idea in 60 seconds

The whole program is built from three kinds of objects that each have exactly one job. Once you understand these three words, you understand the app:

A generator produces objects, a model guarantees those objects are valid, and a repository writes them to the database and returns the real IDs Postgres assigned. The orchestrator is the conductor that calls them all in the correct sequence inside a single transaction.

6. What each Python file does

Top-level files

__main__.py

A 6-line launcher so you can run python -m hospital_datagen. It just imports the Typer app from cli.py and calls it.

cli.py — the front door

Defines the seed command and all its flags (--patients, --last-90-days, --dry-run…). It loads Settings, blends in your flags to build a SeedPlan (how many of each), opens the connection, asks the factory to build the orchestrator, and runs it. It also catches each kind of error and turns it into a distinct exit code (2 = config, 3 = database, 4 = generation, 5 = dependency).

config.py — every knob in one place

A Pydantic-Settings class holding the DB connection details, row counts, the time window, the random seed, etc. Values come from (highest priority first): CLI flags → environment variables → .env file → built-in defaults. It also builds the dsn connection string.

factory.py — the assembler (dependency injection)

Creates one seeded Faker and one random generator, then builds the bundle of all 8 generators and all 8 repositories, and finally a ready-to-run SeedOrchestrator. Keeping all the "wiring" here is what makes cli.py thin and lets tests build pieces in isolation.

orchestrator.py — the conductor

The heart of the app. Runs the 8 steps in FK order inside one transaction, passing each table's returned rows into the next generator. Holds the small data classes SeedPlan (how many to make) and SeedResult (how many were made). For a dry run it raises a private signal to force a rollback.

exceptions.py — one error family

Defines DataGenError and its children (ConfigError, DatabaseError, DuplicateValueError, GenerationError, DependencyError…). One base class lets the CLI catch every domain error in one place and map it to an exit code.

logging_config.py — readable logs

Configures structured logging to stdout at the chosen level.

The models/ folder — validated shapes

Each table has two Pydantic models. This pairing is the clever bit:

  • XxxInsert — the fields we generate and send to the database (no ID yet).
  • XxxRow — XxxInsert plus the primary key Postgres assigned. This is what comes back and gets passed to the next table.

base.py holds the shared RowModel (frozen, rejects unknown fields, trims whitespace) and the allowed-value lists (GENDERS, APPOINTMENT_STATUSES, PAST_STATUSES) that mirror the database's CHECK constraints. Files: department · specialty · medication · doctor · patient · appointment · prescription · prescription_item.

The db/ folder — talking to Postgres

connection.py

ConnectionManager owns a single connection (autocommit off) and exposes a transaction() context manager: it commits if the block succeeds, or rolls back and re-raises a clean domain error if anything fails.

repositories.py

A generic BaseRepository builds safe parameterised INSERT … RETURNING SQL (identifiers composed with psycopg.sql, never string-glued), inserts in batches, and rebuilds the returned …Row. Each table subclass just declares its table, insert_columns, and returning_columns. It also offers existing_values() to read values already in the DB (seed rows / prior runs) so unique columns never clash. Repositories know SQL but nothing about Faker, and never open their own connection.

The generators/ folder — inventing the data

base.py

BaseGenerator gives every generator the same seeded Faker + RNG and a single fixed reference_time ("now"), so a run is fully reproducible and time logic never secretly calls datetime.now(). Also provides truncate() to fit text into column limits.

pools.py

Curated, realistic value lists — real department names, medication names & forms, appointment reasons, dosage patterns. This is why the data looks plausible instead of random gibberish.

uniqueness.py

UniqueValuePool guarantees distinct values within a run. You hand it a "factory" that proposes a candidate; it retries (e.g. appending a number) until it finds an unseen one, or fails loudly with UniquenessExhaustedError. It can be pre-loaded with values already in the DB.

The 8 entity generators

Each produces a list of …Insert objects:

  • department · specialty · medication — pick unique names from the pools.
  • doctor — picks a random existing department + specialty.
  • patient — Faker name, a unique name-derived email, realistic DOB, gender.
  • appointment — owns the time logic: spreads visits across the window, makes past ones completed/cancelled/no_show and future ones scheduled, and bills a cost only for completed ones.
  • prescription — only for completed, past appointments; issued_at is after the visit and never in the future.
  • prescription_item — samples 1..max distinct medications per prescription so the composite PK never collides.

7. How the files connect (with code)

Let's trace one real path through the code so the connections are concrete. Follow the patients table.

1 The orchestrator asks the patient generator to invent rows, pre-loading emails already in the DB so nothing collides (orchestrator.py):

# 5: patients (independent); reserve existing emails (UNIQUE).
patients = self._repos.patients.insert_many(
    conn,
    self._gens.patients.generate(
        plan.patients,
        existing_emails=self._repos.patients.existing_values(conn, "email"),
    ),
)

2 The generator builds validated PatientInsert objects using the shared Faker and the unique-email pool (generators/patient.py):

def generate(self, n, *, existing_emails=()):
    email_pool = UniqueValuePool(reserved=existing_emails)
    return [self._one(email_pool) for _ in range(n)]

def _one(self, email_pool):
    first = self.truncate(self._faker.first_name(), _NAME_MAX)
    last  = self.truncate(self._faker.last_name(), _NAME_MAX)
    email = email_pool.next(lambda attempt: self._email_for(first, last, attempt))
    return PatientInsert(            # <-- validated here; bad data can't pass
        first_name=first, last_name=last,
        dob=self._faker.date_of_birth(maximum_age=95),
        gender=self._rng.choice(GENDERS),
        phone=phone, email=email,
    )

3 The repository writes them and returns real IDs. The base class declares the columns; the SQL is generic (db/repositories.py):

class PatientRepository(BaseRepository[PatientInsert, PatientRow]):
    table = "patients"
    insert_columns = ("first_name", "last_name", "dob", "gender", "phone", "email")
    returning_columns = ("patient_id",)   # <-- Postgres tells us the new ID
    row_model = PatientRow

4 Those returned PatientRow objects (now with patient_id) are passed straight into the appointment generator, which picks a random patient for each visit (generators/appointment.py):

appointments = self._repos.appointments.insert_many(
    conn,
    self._gens.appointments.generate(plan.appointments, patients=patients, doctors=doctors),
)
# inside the generator:
patient_id=self._rng.choice(patients).patient_id   # guaranteed to exist — it was just inserted
That's the whole pattern, repeated 8 times: the Row objects returned from step N become the parents the generator for step N+1 references. Validation (models) and SQL (repositories) stay completely separate from invention (generators), and the orchestrator is the only file that knows the order.

9. How to run it

From the repo root, make sure Postgres is up and the hospital schema is applied, then:

# on the host
cd data-generator
make install                 # poetry install
make run-dry                 # generate + validate against the live schema, then ROLL BACK
make run                     # seed with the default counts

# custom amounts
poetry run hospital-datagen seed --last-90-days --patients 500 --appointments 1200

# or via Docker (same network as Postgres)
make datagen ARGS="seed --last-30-days"
Tip — always try --dry-run first. It runs the entire pipeline, validates every row against the real schema, prints the counts it would insert, and then rolls back so the database is untouched. It's the safest way to confirm everything works.
